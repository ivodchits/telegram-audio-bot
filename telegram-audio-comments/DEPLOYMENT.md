# Deployment

Run the bot on a Linux host with Docker + expose it publicly over HTTPS via
Tailscale Funnel. Survives reboots and is portable to another machine.

Target host so far: Fedora 44, Docker 29 installed and enabled at boot.

## Architecture in one line

```
Telegram users ── HTTPS ──▶ Tailscale Funnel ──▶ tailscaled (host) ──▶ 127.0.0.1:8080 (Docker) ──▶ bot
```

The Docker container's port 8080 is bound to **loopback only**, so the bot is
not reachable on the LAN — Tailscale on the host is the only thing that proxies
public traffic to it.

## What you need before you start

You only have to retrieve/create two values. Everything else has defaults.

### 1. `BOT_TOKEN`

From @BotFather in Telegram:

1. Message **@BotFather**.
2. `/newbot`.
3. Pick a **display name** (shown in chats, e.g. `Audio Comments`).
4. Pick a **username** ending in `bot`, globally unique
   (e.g. `igor_audio_comments_bot`).
5. Copy the token BotFather returns:
   `7654321:AAHxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`.

This goes into `.env` as `BOT_TOKEN=...`.

**Nothing else needs to be configured in BotFather.** The bot opens the Mini App
via inline-keyboard `WebAppInfo` buttons at message-send time, so there is no
required Menu Button URL, no `/setdomain`, no `/newapp`. Optional polish:
`/setdescription`, `/setabouttext`, `/setuserpic`, `/setcommands`.

### 2. `BASE_URL`

Generated by Tailscale Funnel (see step 4 below). Format:
`https://<machine>.<tailnet>.ts.net` — e.g. `https://audiobot.tail1a2b3.ts.net`.
You won't know it until Tailscale is signed in.

### What you do NOT need

- A domain name.
- A public IP / port forwarding on your router.
- A Cloudflare account.
- ngrok.
- A second device on Tailscale — the onboarding wizard will suggest one,
  ignore it. A single-node tailnet is fine for Funnel.

## One-time setup

### Step 1 — Tailscale on the host

```bash
sudo dnf config-manager addrepo --from-repofile=https://pkgs.tailscale.com/stable/fedora/tailscale.repo
sudo dnf install -y tailscale
sudo systemctl enable --now tailscaled
sudo tailscale up
```

`tailscale up` prints a sign-in URL. Open it, authenticate, the machine joins
your tailnet. Pick a clean hostname (e.g. `audiobot`) in the admin console —
the FQDN becomes `audiobot.<tailnet>.ts.net`. Note your tailnet name now:

```bash
tailscale status --self
```

### Step 2 — Enable HTTPS + Funnel in the Tailscale admin console

One-time, per tailnet:

1. https://login.tailscale.com/admin/dns → **Enable HTTPS** (provisions
   LetsEncrypt certs on `*.ts.net`).
2. https://login.tailscale.com/admin/machines → click the machine → **Edit /
   Enable Funnel**. (Or add a `nodeAttrs` rule with `"attr": ["funnel"]` in the
   ACL editor — UI varies.)

### Step 3 — Fill in `.env`

```bash
cd telegram-audio-comments
```

`.env` already exists (copied from `.env.example`). Edit it:

- `BOT_TOKEN=` — paste the BotFather token.
- Leave `BASE_URL=` as the placeholder for now (filled in next step).

Everything else is commented-out defaults; leave them alone unless tuning.

**Never** set `DEV_MODE=true` in this `.env` — it bypasses Mini App auth, which
is the only thing gating session access once the URL is publicly reachable.

### Step 4 — Start the bot

```bash
docker compose up -d --build
docker compose logs -f bot
```

Look for `Mini App URL: https://your-domain.com/app` and no auth errors. (URL
is wrong because `BASE_URL` isn't set yet — that's fine, we fix it next.)

### Step 5 — Turn on the Funnel

```bash
sudo tailscale serve --bg --https=443 http://127.0.0.1:8080
sudo tailscale funnel --bg 443 on
tailscale funnel status
```

`funnel status` prints the public URL. Both commands persist via
`tailscaled.state` — they come back automatically on reboot, no custom systemd
unit needed.

Sanity check (may take up to ~60s on the first hit while certs provision):

```bash
curl -I https://audiobot.<tailnet>.ts.net/healthz
# → HTTP/2 200
```

### Step 6 — Set `BASE_URL` in `.env` and restart

Edit `.env`:

```
BASE_URL=https://audiobot.<tailnet>.ts.net
```

No trailing slash. Must match exactly — the bot's CORS middleware compares
against this string.

```bash
docker compose restart bot
docker compose logs -f bot   # verify Mini App URL line now shows correct URL
```

### Step 7 — Smoke test from Telegram

1. Open a chat with your bot.
2. Send `/start`. The bot should reply.
3. Forward any voice message to it.
4. Tap **Comment on Audio** — the Mini App should open inside Telegram.
5. Record a comment, stop, tap **Stitch & Send** — you should get an OGG back.

If the Mini App opens but fails to load audio or "Forbidden" errors appear in
the in-app browser console: `BASE_URL` is wrong in `.env` (CORS mismatch).

## Reboot behavior (already handled)

Nothing for you to do; documented for confidence:

- `tailscaled` is a systemd unit, enabled at boot.
- `tailscale serve` + `tailscale funnel` state is in `tailscaled.state` and
  restored on daemon start.
- Docker is enabled at boot.
- The bot container has `restart: unless-stopped`, so the daemon starts it.

Verify after a reboot:

```bash
systemctl status tailscaled docker
docker compose ps               # from telegram-audio-comments/
tailscale funnel status
curl -I https://<your-fqdn>/healthz
```

## Moving to another machine

The setup is designed to be portable. On the new host:

### 1. Install Tailscale and reuse the hostname

```bash
sudo dnf install -y tailscale     # or the equivalent for that OS
sudo systemctl enable --now tailscaled
sudo tailscale up --hostname=audiobot
```

In the Tailscale admin console, **remove the old machine** first (or rename
it) so the new one can take the `audiobot` hostname. **Same FQDN means
`BASE_URL` doesn't change and BotFather doesn't need updating.**

Re-enable Funnel on the new machine in the admin console (per-machine
setting, not tailnet-wide).

### 2. Install Docker

Distro-dependent. On Fedora:
`sudo dnf install -y docker-ce && sudo systemctl enable --now docker`.

### 3. Copy the project folder

From the old host:

```bash
# On old host
cd ~/repos/telegram-audio-bot
tar czf telegram-audio-comments.tgz telegram-audio-comments/
scp telegram-audio-comments.tgz newhost:~/
```

```bash
# On new host
tar xzf telegram-audio-comments.tgz
cd telegram-audio-comments
```

The tarball includes `.env` (with your token) and `data/` (sessions + audio).
**Treat it like a secret** — `.env` has your bot token.

### 4. Start everything

```bash
docker compose up -d --build
sudo tailscale serve --bg --https=443 http://127.0.0.1:8080
sudo tailscale funnel --bg 443 on
curl -I https://audiobot.<tailnet>.ts.net/healthz
```

### 5. Shut down the old host

Once smoke-tested, stop the bot on the old machine
(`docker compose down`) so two bots don't both try to poll Telegram.

## Operations cheat sheet

```bash
# From telegram-audio-comments/

docker compose ps                  # is it running?
docker compose logs -f bot         # tail logs
docker compose restart bot         # restart after editing .env
docker compose down                # stop
docker compose up -d --build       # apply code changes
docker compose pull && docker compose up -d   # apply base-image updates

# Tailscale
tailscale status                   # tailnet view
tailscale funnel status            # current funnel mappings + public URL
sudo tailscale funnel --bg 443 off # stop exposing publicly
sudo tailscale serve reset         # clear all serve/funnel config

# Disk usage
du -sh data/                       # session + audio storage
```

Sessions auto-expire after `SESSION_TTL_DAYS` (default 7) — cleanup runs
hourly inside the bot, see `cleanup.py`.

## Troubleshooting

**Mini App opens but API calls return 403 / "Forbidden"**
`BASE_URL` in `.env` doesn't match the public URL. Check for trailing slash,
scheme, host. Restart the bot after fixing.

**Mini App opens but audio doesn't load**
Same root cause — CORS rejects the `<audio>` request. Check
`docker compose logs bot` for auth-middleware warnings.

**`tailscale funnel status` shows no mapping**
`tailscale serve` was never set up, or `tailscaled.state` got wiped. Re-run
the two `serve` / `funnel` commands.

**TLS cert errors on first request**
Wait ~60s and retry. LetsEncrypt provisioning is lazy on first request.

**Bot doesn't respond at all in Telegram**
Either `BOT_TOKEN` is wrong (check logs for `Unauthorized`) or two instances
are polling concurrently (e.g. old host wasn't shut down). Telegram returns
409 Conflict in that case.

**"address already in use" on `docker compose up`**
Something else is on port 8080. Either kill it or set `PORT=8081` in `.env`
(and update the `tailscale serve` target to `http://127.0.0.1:8081`).

**Want to roll back to a previous version**
`docker compose down`, `git checkout <ref>`, `docker compose up -d --build`.
`data/` is untouched.

## Switching to webhook mode (optional, later)

Polling is simpler and works fine over Funnel. If you want webhooks (lower
latency, fewer outbound long-polls):

1. Generate two secrets:
   ```bash
   python -c 'import secrets; print(secrets.token_hex(16))'  # path suffix
   python -c 'import secrets; print(secrets.token_hex(32))'  # header token
   ```
2. In `.env`:
   ```
   MODE=webhook
   WEBHOOK_SECRET_PATH=/tg/<first-secret>
   WEBHOOK_SECRET_TOKEN=<second-secret>
   ```
3. `docker compose restart bot`. The bot calls `setWebhook` against
   `BASE_URL + WEBHOOK_SECRET_PATH` on startup.

Funnel already exposes `/tg/...` because it proxies the whole host — no
extra config needed.

## Security notes

- `.env` contains the bot token — back it up alongside `data/`, but never
  commit it.
- The host port binding `127.0.0.1:8080` means LAN devices cannot reach the
  bot directly. Everything has to go through Tailscale.
- Funnel makes `https://<fqdn>.ts.net` publicly reachable. The bot's
  `auth_middleware` (Telegram initData HMAC verification) is the only thing
  protecting session data — never set `DEV_MODE=true`.
- Session IDs are 22-char `secrets.token_urlsafe(16)` — unguessable, but
  treat the URL as sensitive anyway.
